# billcheck V0.1 — design notes (living)

> **Status: LEADING (V0.1).** Decisions from the V0.1 brainstorm, captured as we go.
> Not gospel — revised as we learn. Entry point: [START-HERE.md](START-HERE.md). _Updated 2026-06-16._

## Framing
billcheck V0.1 is a **helpful, chat-first medical-bill advisor** — an agent whose job
is to give the user the most useful advice, backed by broad knowledge and a lot of
tools. The deterministic engine is **one tool among many** (used when a number or
verdict must be trustworthy), not the spine. Most of the value — recognizing the
situation, orienting the user, the right lever, drafting, answering — is advice/action
that never touches the engine. (Supersedes RETHINK's "shell around a deterministic
core" wording.)

**Bright line (unchanged):** the agent converses, gathers context, picks tools, and
explains; it never originates a dollar amount or a verdict — every number/verdict a
user sees or a letter contains traces to a deterministic source.

## Data model — three levels
```
Case   (the episode — e.g. "my knee surgery")        ← the "home" unit; status is DERIVED
 └─ Bill   (one biller's charge for the episode)      ← carries the lifecycle + status; actions target this
     └─ Documents  (statement · itemized bill · EOB(s) · receipt · letter · …)  ← typed evidence
```
- **Invariant:** a bill's documents (its statement, itemized bill, EOB(s), receipts,
  dispute letter) are bound to that one bill in one case — never split across cases.
- **Linking is agent-driven, user-confirmable** (not a rigid key): the agent proposes
  "this looks like the EOB for last week's surgery bill — file it together?"; a
  provider + account + date-of-service heuristic seeds the suggestion.
- **One bill → multiple EOBs** is normal: split/partial adjudication, or coordination
  of benefits (primary + secondary insurer EOBs for the same services).

## Vocabulary (most-common patient-facing terms)
- **statement** — the early, *unofficial* provider document: balance due, summary or
  department-level charges, **no line items**.
- **itemized bill** — the *official*, line-by-line bill (codes, units, per-line price);
  hospital standardized form = UB-04, professional = CMS-1500. Colloquially "the bill."
- **EOB (Explanation of Benefits)** — the insurer's document ("this is not a bill");
  Medicare's version = MSN.
- Others (unambiguous): receipt · collection notice · denial letter · GFE (Good Faith Estimate).
- Naming wrinkle: the model *object* is also called a "bill" (one biller's charge), while
  "the bill" as a *document* = the itemized one. Keeping the double-use for now; rename
  the object (e.g. "charge"/"claim") if it causes confusion.

## Bill lifecycle (a backbone, not a rigid corridor)
**New → Gathering → Reviewed → Acting → Resolved** (+ Closed)
- **New** — just captured; figuring out what it is.
- **Gathering** — need more to judge (request the itemized bill; wait on the EOB; a triage question).
- **Reviewed** — assessed: audit / verdict / advice ready.
- **Acting** — a play is in motion (dispute / appeal / FAP / validation sent; awaiting response).
- **Resolved** — settled (paid-as-correct, reduced, written off, won); **Closed** — user dropped it.
- The agent places a bill wherever reality is — it can jump ahead (straight to "in
  collections") or back (a corrected statement arrives). Evolved from V0's `CaseState` enum (reuse).

## Case status — a derived urgency rollup
- Computed **deterministically** from the bills' (lifecycle stage + "ball in whose court?").
- Priority ladder, most-urgent wins: **Action needed → In progress → Waiting on insurer → Resolved.**
- "Resolved" requires **all** bills resolved.
- The chip shows the mix, not a lossy word: e.g. *"⚠ Action needed · 1 of 3 (2 resolved)."*
- The agent **narrates** it; it does not decide it.

## Updates from case studies (2026-06-17)
Real cases (see [v0.1-cases.md](v0.1-cases.md)) revised a few things:
- **Lifecycle gains a front state + a loop:** **Expected (pre-bill)** → New → Gathering →
  Reviewed → Acting → Resolved (+ Closed; can **Reopen**). *Expected* = tracking a charge that
  hasn't arrived (scheduled service, a GFE in hand) — needed for "home for your billing."
  **Acting** is often a **multi-step campaign** (an escalation ladder), not a single act.
- **The product is a persistent advocate, not a one-shot auditor** — cases run for weeks across
  channels with deadlines + automated follow-up (durable-agent shape). 2 of the first 3 cases
  never touch the engine.
- **Levers are an arsenal, and exist for almost any bill — even "valid" ones:** benchmark-vs-market
  (à la Granted) · negotiate · payment plan · charity care/FAP · dispute/appeal · FDCPA ·
  **press/public** · hardship/bankruptcy. Honest value = a *ranked options menu + the real odds*,
  never "nothing." (Temper expectations without defeatism.)
- **Press/public pressure is a real, productized lever:** a public showcase ("wall," renamed to
  avoid lawsuits) + active outreach to local journalists/consumer reporters for egregious,
  documented, sympathetic cases. Differentiator; needs legal guardrails (truthful, user's own
  story, provider right-of-reply, opt-in).
- **The activity log is the asset** — verbatim communications + the user's narrative + decisions +
  deadlines = the evidence chain a chargeback/regulator/court needs. Capture everything, timestamped.
- **"Amounts," not "ledger"** for the money breakdown (billed / allowed / paid / owed / in-collections / disputed).
- **Scope is wider than coding audits:** coding errors → surprise/affordability → service
  misrepresentation/non-delivery disputes. Through-line: "help me with this bill."

## Triage is the center of gravity (2026-06-17, Pedro)
The case corpus is journalism-sourced, so it **over-samples egregious disputes and the "press won
it" ending** — a selection artifact, not the real distribution (see [SYNTHESIS](v0.1-cases/SYNTHESIS.md)
caveat). The true **highest-frequency, highest-value jobs are mundane triage**:
1. **"Don't pay that yet — it's a *statement*, not the final bill."** (wait for the itemized bill / EOB)
2. **"This looks correct, here's why — you're fine to pay it."** (reassurance + permission)
Design implication: optimize the **common path first** (any-shape input → coverage situation +
**document-type detection** → one of a few common verdicts, incl. "don't pay yet" / "looks fine"),
and treat the full dispute arsenal (incl. press) as **depth behind it**. The dangerous error is a
**false "pay it,"** so triage accuracy — especially statement-vs-itemized-vs-EOB — is the first
thing to get right and to eval.

## Round-2 decisions (Pedro, 2026-06-17)
1. **MCP — explicitly OUT of V0.1 (revised 2026-06-17).** The idea: exposing billcheck inside
   ChatGPT/Claude could answer "why not just use a general model?" (moat = tools + data/system-of-record +
   durable state + bright line). But Pedro: *"if thinking of MCP makes V0.1 even a little harder, leave it
   out — get the app right, don't build for a future that may never come."* So it's out; must not add any
   complexity now. **The one principle it had motivated survives on its own merits:** keep the **bright
   line + guards inside the tools** (not just the prompt) — that's required for our own app's safety +
   testability regardless of MCP. → [03](v0.1-design/03-agent-loop-and-tools.md), [04](v0.1-design/04-chat-ux.md).
2. **V0.1 is mobile-first** — responsive web, **not** a native app (snap-a-photo upload is inherently
   mobile; no app-store friction). Don't invest in desktop polish; it comes ~free via responsive layout.
3. **Voice = speech-to-text *input* only** (dictate instead of type), not a 2-way voice agent.
   Cheap/free (the mobile keyboard's dictation already works in any text field; Web Speech API / Whisper
   for in-app). Defer until text works on its own.
4. **Cards: keep simple, ~4–5 to start** (Pedro: "can prob be more than just 2"). The set: **VerdictCard,
   AmountsPanel, DocChip+Viewer, ConfirmButtons, + a light ActivityLog.** Intake stays **conversational**
   (stub the mini-form). Owned/accessible/text-fallback each. Grow from there.
5. **Orchestrator voice principle:** insightful, direct, **concise — don't over-explain.** Offer a
   lightweight **"explain why / say more / in detail"** affordance (a follow-up chip, or the user just
   asks). Pairs with the conversation-trap / cognitive-load research and the bright line (concise +
   provenance). → [03-agent-loop-and-tools](v0.1-design/03-agent-loop-and-tools.md) system prompt.
6. **Build V0.1 *now*; the June 27 hackathon is a checkpoint, not the target.** Coding is fast with
   current models; **testing is the bottleneck** → invest in eval + **user-simulation** early. Docs
   de-anchored from the date (`PLAN.md`, `06-v0.1-scope.md`).
7. **Agent-native architecture research folded in** (→ Q3 + [research](research/2026-06-17-agent-architecture-best-practices.md)):
   single-orchestrator (not multi-agent) is right; **enforce the bright line structurally at the artifact
   boundary** (not prose); **deterministic pipeline (parse→audit→fact-gen), open-ended conversation**;
   external/path-addressable/compaction-stable case memory; gather→act→**verify** loop; package procedures
   as **Agent Skills**; minimal-viable tools returning human-readable ID-bearing facts; **trajectory +
   bright-line evals** from day one. _(Testing/user-simulation research still running → Q7.)_

## Round-3 decision (Pedro, 2026-06-17): the orchestrator reasons like a human expert
**Don't lock the agent into a fixed set of outcomes.** For each decision it should think like a sharp
patient-advocate and say what it would say — if that doesn't match a predefined verdict, **"other →
tailored advice" is valid.** Be **confidence-aware**: offer 2 options with pros/cons (or surface
uncertainty) on a real fork; don't fake certainty. **Lead with the user's actual situation, not a reflex.**
_Two Chairs lesson:_ charged $179 with no invoice → "don't pay yet, it's a statement" is wrong and "ask
for the itemized bill" alone is unhelpful; the expert opening **names it** ("you were charged and you're
surprised"). The verdict taxonomy = **recognized patterns/priors, not a switch statement.** This is **safe
because of the bright line** (open-ended *advice*; deterministic *numbers*). → Q2 + Q3 updated.

## DECISION — pure greenfield (Pedro, 2026-06-17, comment 10)
**Plan and build V0.1 as if V0 did not exist.** Concern: anchoring on V0 code bends a chat-first agentic
advisor toward the old linear-audit-funnel shape, and **adapting 95%-fit code quietly costs the 5% that
mattered.** So: **treat ALL V0 code — including the engine — as at most a reference, never a foundation;
never adapt-to-fit; design every part on its own merits for the new architecture + the latest model.**
- The reuse-inventory ([00](v0.1-design/00-v0-reuse-inventory.md)) is **demoted to historical reference**
  ("V0 existed and worked; we are NOT building on it").
- **What carries over is the *acceptance bar*, not the code:** the validation philosophy + the documented
  test cases (the 31 cases; the engine golden fixtures' properties — hand-computed/anti-circular,
  injection-resilience, reproducibility) become the bar the **rebuilt** engine/letters/safety logic must
  pass — re-expressed for the new design. (Keeping *tests* constrains correctness without anchoring design.)
- **Upside:** pure greenfield **dissolves review P0-1 and P0-2** — there's no "second client bypassing the
  shared client" (we design one agent-loop client with the guards baked in) and no "loosen the legacy
  state trigger" (the living-thread model is native in a fresh schema). P0-3 (structural bright line) stands.
- **Trade-off Pedro accepted (I flagged it):** re-deriving + re-validating the engine's coding logic, the
  letter bright line, and PHI/RLS from scratch is real effort/risk → which makes the **testing/simulation
  workstream (comment 8 / Q7) even more load-bearing.** Build the safety gates first.
